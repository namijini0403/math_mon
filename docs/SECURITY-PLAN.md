# DRACONIS 보안·개인정보 보호 계획 (감사 2026-06-13)

초등학생 대상 학교 앱이므로 **개인정보 최소수집 + 식별정보 외부 미전송**이 최우선 원칙.
이 문서는 ① 현재 잘 되어 있는 점 ② 위험 지점(우선순위별) ③ 보완 로드맵 ④ 운영 전 법적 체크리스트.

> 비전공 운영자(교사)를 위한 요약: 큰 그림(애들 식별정보를 외부 분석 서버로 안 보낸다)은
> 코드에서 실제로 지켜지고 있어요. 아래는 "더 단단하게" 만들기 위한 보완 목록이에요.

---

## 1. 이미 잘 되어 있는 점 (설계 강점)

- **운영 DB / 분석 DB 물리적 분리** (`db.ts`의 `pool` vs `analyticsPool`, 서로 다른 Supabase 프로젝트). 코드에서 혼용 금지.
- **분석 DB엔 가명(pseudonym_id)만** + **이벤트 화이트리스트 이중 검증**(클라 `analytics/types.ts` + 서버 `index.ts sanitizeEvent`). 닉네임·자유텍스트·실명 등 식별정보가 이벤트에 **구조적으로 들어갈 수 없음**.
- **`ANALYTICS_DB_URL` 미설정 = no-op** — 동의 절차 전 수집을 원천 차단하는 법적 안전장치.
- **PIN은 scrypt 해시**(timing-safe 검증), 평문 저장 안 함. JWT는 HS256 timing-safe 검증, access(메모리)·refresh 분리.
- **모든 SQL이 파라미터 바인딩**($1, $2 …) — SQL 인젝션 방어.
- **서버 저장 save 블롭에서 식별자 제외**(`api.ts SAVE_KEYS` — nickname/classCode/studentId 미포함).

---

## 2. 위험 지점 (우선순위별)

### 🔴 HIGH

**H1. PIN 무차별 대입(brute-force) 방어 없음** — ✅ **보완 완료(2026-06-13)**
- 4자리 PIN(1만 가지) + `/api/auth/join`에 시도 제한 없어 같은 반 친구 계정 탈취 가능했음.
- 적용: (classCode, nickname)별 **인메모리 카운터 — 15분 내 10회 실패 시 15분 잠금**(429). 정상 사용자엔 영향 없을 만큼 관대하고, **내부 오류 시 fail-open(로그인 우선)**. 성공 시 카운터 초기화. (분산/영속 rate-limit은 추후 — 단일 인스턴스 가정.)

**H2. 공유 기기 localStorage 잔존 (학교 태블릿 시나리오)** — ✅ **보완 완료(2026-06-13)**
- 학생 A가 로그아웃 없이 자리를 뜨면 `localStorage['math-mon-save']`(닉네임·반코드·studentId)와 refresh 토큰(`draconis-refresh`)이 남아 다음 학생 B가 A의 계정·기록에 접근하는 위험.
- 적용: 프로필에 **"다른 친구로 바꾸기(로그아웃)"** 버튼 추가. `clearAuth()`(메모리 access 토큰 + localStorage·IndexedDB refresh 토큰 제거) + `resetAll()`(로컬 게임상태·식별자 초기화) → 로그인 화면. 등록 학생은 서버 세이브로 재로그인 시 복원. **로그인 절차 자체는 그대로(마찰 없음).**

### 🟠 MEDIUM

**M1. `/api/progress` 비인증 경로(하위호환)** — ✅ **보완 완료(2026-06-13)**
- Bearer 토큰 없으면 `studentId`만으로 진도 덮어쓰기 허용했음.
- 적용: **Bearer 토큰 필수화** + 토큰 `sid`와 `studentId` 일치 검증. 토큰 없으면 401(클라는 로컬 저장으로 graceful 폴백, 로그인 무관).

**M2. 교사 API 키가 URL 쿼리 + 단일 공유키** — 🟡 **부분 보완(2026-06-13)**
- `GET /api/teacher/:code?key=...` → 키가 접근 로그에 남던 문제.
- 적용: 키를 **`X-Teacher-Key` 헤더로 이동**(쿼리 로그 노출 제거, TeacherPage도 갱신). **잔여**: 교사별 키 발급·회전(여전히 단일 공유키).

**M3. refresh 토큰 localStorage 저장 (XSS 노출면)**
- XSS 취약점 발생 시 refresh 토큰(90일) 탈취 → 장기 세션 가로채기.
- 보완: 의존성 최소화 + CSP로 XSS 표면 축소. 배포 구조가 허용하면 httpOnly 쿠키 검토.

**M4. 보안 헤더 부재**
- CSP, X-Content-Type-Options, Referrer-Policy, X-Frame-Options, HSTS 등 미설정.
- 보완: Hono 미들웨어로 기본 보안 헤더 추가(즉시 가능, 비파괴적). CSP는 스크립트 출처 제한으로 XSS 완화.

### 🟡 LOW

- **L1. `/api/events` rate limit 없음** — 토큰 보유자가 분석 DB에 스팸 가능(요청당 100개 제한은 있으나 반복 무제한). 토큰/IP당 분당 제한 권장.
- **L2. 닉네임에 실명 입력 가능** — 안내만 있고 검증 없음. 운영 DB에 실명 저장 위험. 입력 시 소프트 경고 + 교사 안내.
- **L3. JWT_SECRET 미설정 시 랜덤(가용성)** — 운영서 미설정 시 재시작마다 세션 무효. 배포 체크리스트로 강제.
- **L4. DB 연결 SSL 명시 확인** — 연결 문자열에 `sslmode=require` 포함 확인(Supabase 기본 요구).

## 3. 보완 로드맵 (단계별)

| 단계 | 항목 | 난이도 | 비고 |
|---|---|---|---|
| **1단계 (즉시·비파괴)** | M4 보안 헤더, H2 로그아웃, M2 키 헤더 이동, L2 닉네임 소프트검증, L3/L4 배포 체크리스트 | 낮음 | 동작에 영향 적음 |
| **2단계 (핵심 방어)** | H1 PIN rate-limit/잠금, M1 Bearer 필수화 | 중간 | 서버 상태/테이블 추가 |
| **3단계 (심화)** | M3 CSP 강화·쿠키 검토, L1 events rate-limit, `npm audit` 정례화·의존성 최소화 | 중상 | 점진 적용 |

각 단계는 독립적으로 배포 가능. 1단계는 사용자 동의 없이도 안전하게 적용(개인정보 처리 변화 없음, 방어만 강화).
