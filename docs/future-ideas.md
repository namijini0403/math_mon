아직 구현하지 않은 확장 아이디어 모음. 기술 가능성·제약·기존 코드 연결점·추천안을 정리해
나중에 착수할 때 바로 설계로 넘어갈 수 있게 한다. (작성 2026-06-13)

## 1. 위치/AR 히든 미션 — "운동장 특정 장소에 가면 깜짝 문제 + 이벤트 카드"

학교 운동장의 특정 지점에 가면 히든 수학 문제가 나오고, 풀면 이벤트(보물) 카드가 드롭.

### 기존 코드 연결점 (붙일 자리 이미 있음)
- `client/src/game/hiddenMissions.ts` — "조건이 갖춰지는 순간 갑자기 보물 카드"가 나오는 메커니즘이 이미 있음.
  여기에 "특정 spot 방문" 플래그(`visitedSpots: string[]`)만 추가하면 코드 변경 최소로 연결됨.
- 보물 카드 드롭 경로는 이미 "히든 미션"이 포함되어 있어 게임 경제(카드 남발 방지) 결정과도 일관(§5).
- 문제는 기존 파라메트릭 생성기 재사용 → spot마다 `#/spot/:id` 라우트로 히든 스테이지 진입.

### 구현 방식 (견고함 순)
1. **QR 마커 (추천)** — 운동장 지점(미끄럼틀·등나무·국기게양대 등)에 QR판 부착.
   스캔 → `#/spot/<id>` → 히든 문제 → 풀면 이벤트 카드.
   - 위치 정확도 100%, iOS/안드로이드 모두 OK, 권한은 카메라만.
   - **위치정보를 수집하지 않아 개인정보 부담이 가장 적음** → 학교 도입에 최적.
2. **GPS 지오펜스** — `navigator.geolocation`으로 좌표 반경 N m 진입 시 문제 등장.
   진짜 "그 장소에 가면" 느낌이지만 **실외 GPS 오차 5~15 m**라 좁은 운동장에선 부정확할 수 있음.
   미성년자 위치정보 수집 = 동의·처리방침 필요.
3. **WebXR 3D AR (멋 부리기 단계)** — 카메라 화면에 드래곤/문제가 떠 보임.
   - 안드로이드 Chrome 지원, **iOS Safari는 WebXR AR 미지원**(아이패드 학생 많으면 큰 제약).
   - AR.js 이미지/마커 방식으로 우회 가능.

### 학교 도입 고려사항
- 🔒 개인정보: 미성년자 위치정보는 민감(LA 동의 맥락과 동일). ①QR 방식은 이 부담이 없음.
- 📱 기기 편차: 저가 안드로이드·구형 아이패드 → ①이 호환성 최고.
- 🚸 안전: 학생이 화면 보며 이동 → "정적인 지점 도착 후 풀기" 방식 권장.

### 추천 1안
**QR 마커 + 기존 히든 미션 확장.** `hiddenMissions`에 spot 방문 플래그 추가 → `#/spot/:id` 히든
라우트 → 파라메트릭 문제 → 이벤트 카드. AR 카메라 오버레이는 이후 ③으로 고도화.

## 2. 걸음수(만보기) 연동 보상 — "하루 3000보 넘으면 배지·아이템"

애들이 몸을 움직이게 하는 게임화. **앱 켠 동안 1일 3000보** 달성 시 배지·드래곤 과일/아이템 지급.
**방향 확정(사용자): 생활 트랙의 첫 기능으로 바로 착수. 운영 전 학부모·학생 동의 철저히 받음.**
궁극 목표는 수학앱 → 학교생활(학습+생활) 전반 앱.

### 시간대 이벤트 게이팅 (사용자 아이디어 — 등교길 운동장 문화)
- **특정 시간에만 이벤트 열기 가능** (예: 평일 07:30~08:40만 "아침 운동장 미션").
  구현: 클라이언트 시계로 시간창 판정(요일+시각) → 조건 충족 시 히든/이벤트 노출.
- 클라 시계는 조작 가능하나 등교 이벤트엔 조작 동기 낮아 로컬시간으로 충분. 엄밀히 하려면 서버시간(`/api`에서 now 제공).
- 위치 히든 미션(§1 QR)과 결합 시 "아침에 운동장 QR 돌기 → 보상" 문화 설계 가능.

### PWA 한계와 극복 (핵심 의사결정)
- **PWA만으론 진짜 백그라운드 만보기 불가**: 웹앱은 포그라운드(화면 켜짐+앱 열림)일 때만
  가속도계 접근. iOS Safari는 백그라운드 측정 원천 차단.
- **극복 = 코드를 버리지 않고 네이티브 껍데기로 감싸기(Capacitor) + OS Health 읽기.**
  - iOS → HealthKit, 안드로이드 → Health Connect. OS가 백그라운드에서 항상 센 값을 **읽기 권한 1개**로 가져옴(흔들기 조작 불가, 앱 안 켜도 누적).
  - 기존 React+Vite 코드 100% 재사용. TWA·Web Periodic Sync는 센서 백그라운드 안 되니 걸음엔 부적합.
- 배포 현실: **안드로이드 우선**(Health Connect 무료, 개발자 등록 ~$25, 내부배포·사이드로드 쉬움).
  iOS는 연 $99·심사·프로비저닝(단 Apple School Manager+MDM이면 학교 일괄 배포).

### 단계적 로드맵 (지금 코드가 헛되지 않게)
```
1단계(지금)  PWA에 걸음수 데이터모델 + 3000보 보상 + UI 골격
            ↳ 측정부를 StepSource 어댑터로 추상화
2단계(전환)  Capacitor 래핑 → 어댑터를 Health Connect/HealthKit로 교체 → 진짜 백그라운드
3단계(배포)  안드로이드 내부배포 먼저 → iOS 이후
```
**1단계를 어댑터 추상화로 잘 넣어두면 2단계는 어댑터만 갈아끼우면 됨. 작업 안 버려짐.**

### 1단계 구체 설계 (PWA, 지금 구현 범위)
- **저장 (store.ts, persist 키 추가만 — §5 호환 유지, 'math-mon-save' 키 불변)**
  ```ts
  interface StepState {
    date: string;          // YYYY-MM-DD, 날짜 바뀌면 today 리셋(rollover)
    today: number;         // 오늘 걸음수
    goalDays: number;      // 3000보 달성 누적일 (배지 조건)
    streak: number;        // 연속 달성일
    lastGoalDate: string;  // 마지막 달성일(연속 판정)
    history?: Record<string, number>; // 최근 N일(선택, 그래프용)
  }
  ```
  액션: `ingestSteps(count)`(어댑터가 호출하는 단일 진입점, rollover 포함),
  5000 돌파 순간 1회 보상 트리거(배지 evaluate + 과일 지급).
- **측정 어댑터 (StepSource 인터페이스 — 교체 가능하게)**
  ```ts
  interface StepSource {
    id: 'foreground' | 'manual' | 'health';
    isAvailable(): boolean;
    requestPermission?(): Promise<boolean>; // iOS DeviceMotion 권한
    start(onUpdate: (todaySteps: number) => void): void;
    stop(): void;
  }
  ```
  - `ForegroundPedometer`: DeviceMotion 가속도 피크검출(앱 켠 동안 체감용, 정확도 대략적).
  - `ManualSource`: 교사 디버그 입력(흐름 검증·시연용, 교사코드 0403 뒤).
  - (2단계) `HealthSource`: Capacitor Health 플러그인 → ingestSteps 연결.
- **보상 (기존 패턴 재사용, 카드는 제외 — §5 카드 남발 방지)**
  - badges.ts `BadgeStats`에 `stepGoalDays` 추가 → 배지 `step-first`(첫 3000보)·`step-7`·`step-30`.
  - 3000보 첫 달성 시 드래곤 과일 N개(드래곤 성장과 연결 → "운동하면 드래곤도 큰다").
- **UI**: 만보기 카드 — 오늘 걸음/5000 원형 게이지, 달성 누적일·연속일, 측정 시작 버튼(권한 요청).
  드래곤 방 위젯 또는 `/steps` 라우트. 홈에 오늘 진행 미리보기.
- **검증**: 피크검출·rollover·5000돌파 보상은 vitest 단위테스트, UI는 수동(데스크톱엔 센서 없음 → ManualSource로 흐름 확인).

### 2단계 구체 (네이티브 전환)
- Capacitor 도입(`cap init`, ios/android 추가, Vite build→www), Health 플러그인(Health Connect/HealthKit) + 권한(manifest/plist).
- `HealthSource` 작성 → `ingestSteps`에 연결. 앱 열 때 OS 오늘자 누적 걸음 read(+주기 동기화).
- 배포: 안드로이드 내부테스트 트랙, 서명키.

### 개인정보 (불변 원칙 — §6 맥락)
- 🔒 걸음수 = 건강정보(위치정보와 동급 민감). **로컬(localStorage)에만 저장, 분석DB 미전송**(화이트리스트 21종 안 건드림 = 동의 전 안전장치 유지).
- 서버/분석 전송이 필요해지면 동의서·처리방침에 "걸음수(건강정보)" 항목 추가 후 별도 설계.

## 3. 정적 이미지 애니메이션 — "Codex 일러스트를 움직이게"

정적 PNG를 움직이는 건 **이미지 생성과 달리 전부 코드 영역** → 메인(Claude)이 직접 가능.
이미 framer-motion으로 드래곤 둥둥·하트 파티클·출석 선물 흔들림 등 구현 중.

| 단계 | 방법 | 담당 | 비고 |
|---|---|---|---|
| ① 통짜 PNG 흔들기 | framer-motion 호흡(scale)·부유(y)·미세 회전 | 메인, 즉시 | 무료 |
| ② 부위 분리 애니 | 날개·꼬리·눈을 투명 PNG 레이어로 분리 → 펄럭·눈깜빡 | 메인 + Codex(레이어 분리) | 살아있는 느낌 ↑ |
| ③ 스프라이트 프레임 | 걷기 3~4프레임 그림 → CSS `steps()` 전환 | Codex(프레임 제작) | 진짜 캐릭터 애니 |
| ④ Rive/Lottie | 전문 벡터 애니 툴(상태머신) | 전문 제작 | 최고급, 현재 오버스펙 |

- **AI(MCP)로 PNG→자동 영상화는 비추**: 현재 환경에 적합 도구 없고, 게임 캐릭터는 통제·일관성이 안 됨. 코드 애니가 정석.
- **Codex 협업 포인트**: ASSET_SPEC에 "②용 — 날개/꼬리/눈을 분리한 투명 PNG 레이어" 또는
  "③용 — idle/걷기 N프레임 시트" 명세를 추가하면 코드 애니로 승급.

## 공통 메모
- 1·2번 모두 **개인정보(미성년 위치·건강정보) 동의 절차**가 기술보다 먼저 풀어야 할 과제.
- 보상 연결부(카드·배지·아이템·과일)는 기존 아키텍처에 자연스럽게 붙음 → 진입장벽은 "데이터 소스"와 "동의·배포 형태"이지 게임 로직이 아님.
- 만보기 1단계는 어댑터 추상화만 지키면 네이티브 전환 시 손실 없음.
